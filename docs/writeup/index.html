Building a Multi-Agent System That Optimizes Vision-Language Models for the Edge

The system is the deliverable. A competitive edge model is just the proof-of-work — and the most useful results were the failures.
Han Wook Cho · June 2026 · source on GitHub

TL;DR

I built an autonomous multi-agent loop that takes a general 3B vision-language model and constructs a right-sized edge student from it — proposing a model spec, assembling it, distilling it from a teacher, evaluating it on the same path as the benchmark, and re-routing the next attempt from the result. Along the way it surfaced a clean, transferable finding: for a small constructed VLM, the dominant lever is matching the training-data distribution to the target benchmark — not the optimizer, the rank, or the budget. The resulting student is edge-viable (runs at ~80 tok/s in ~1.6 GB on Apple Silicon). It is not state-of-the-art, and I'll be precise about why. All benchmark numbers here are internal-only.

Why a system, not a model

Small edge VLMs like LFM2-VL, SmolVLM, MiniCPM-V, and FastVLM are the output of team-months of expert optimization. The bet behind this project is that the process of producing such a model can be automated and compressed — that an agent loop, with a human gating only consequential decisions, can navigate the optimization journey. So the deliverable is the system; a competitive model is how you prove the system works, not the point in itself.

How the loop works

After an early wrong turn — distilling into an already-optimized edge model, which only moved it off its tuned optimum and regressed — the approach became construction: assemble a new, right-sized student from parts (a SigLIP vision encoder + a fresh projector + a small Qwen2.5-0.5B language model) and distill it from a 3B teacher. Crucially, this was made a system capability:

The agent proposes a content-addressed model spec → a deterministic builder assembles, aligns, distills, and evaluates it on the same inference path as the benchmark → the result is recorded to a ledger → the agent reads it and proposes the next spec. The human writes the generic builder once; the agent drives every instance.

The finding: distribution matching is the lever

The student kept failing to learn a second skill. Four times. Adding captions hurt grounding; adding multiple-choice data made the model forget its yes/no grounding while never actually learning multiple-choice. Each looked like "catastrophic forgetting," and the instinct was to fight it with rehearsal and more training.

That instinct was wrong, and the system's own evaluation said so: more training and rehearsal didn't help. The real diagnosis — verified before spending more compute — was a distribution mismatch. The benchmark it floored on (MMBench) is ~70% science/knowledge/reasoning questions; the student only ever trained on everyday-photo (COCO) data. A quick probe confirmed the student could answer multiple-choice questions on in-distribution data — it just couldn't answer MMBench's kind. So I built training data from the matching distribution (ScienceQA), and the benchmark cleared its floor for the first time.

Together with the original result, the lever is proven in both directions: a small student acquires exactly the skill whose data distribution it trains on (everyday photos → object grounding; science MCQ → MMBench) and stays at chance on the rest. The "interference" failures were, underneath, a distribution story all along — the same reason production VLMs train on broad mixtures spanning their eval suites.

The honest boundary — and a methodology catch

Could one student hold both skills? A balanced mix cleared both benchmarks at once — once. Then I discovered the construction runs had no fixed random seed, re-ran the same configuration three times, and the truth came out: one benchmark was robust across seeds, but the other swung wildly (a near-coin-flip). The "one student, two skills" milestone had been a lucky draw. The lesson stuck: single-run results aren't trustworthy; claims have to be multi-seed. Pushing more grounding data shifted the average up but not the variance — which points at model capacity, not data, as the remaining wall at this size.

This is the part I'm most glad I caught. It's easy to ship a shiny single-run number. The variance test turned an over-claimed "milestone" into an honest, sharper result — and that honesty is the whole point of building a system you can trust to steer itself.

Does it actually run on the edge?

The student had only been scored on a Mac. To check the edge half, I ported it to MLX (Apple's on-device runtime) and — before any device-specific work — verified the ported model reproduces the original exactly (the vision tower matched to 1 part in 5,000; the assembled model produced identical output). Only then did I measure performance:

1.6 GB fits an iPhone 16 Pro comfortably — the exact budget that made the 3B teacher (6.5 GB) non-viable on-device. The model the system built is genuinely edge-deployable.

What this is — and what it isn't

It is: a working autonomous construction system; a clean, transferable principle; a disciplined evaluation methodology (floor-adjusted, multi-seed, verify-before-building); an edge-viable, faithfully-reproduced student; and a documented trail of honest negative results that drove every pivot.

It is not: a state-of-the-art model. The student is below the reference models on quality, is per-axis (different students clear different benchmarks), and is variance-limited at this size. A single competitive multi-skill model is gated on capacity — a deliberate next phase, not a method failure.

On the numbers: every benchmark figure here is internal-only — 100-sample slices on a non-official protocol, used to steer experiments on a held-constant path. They are internally consistent and honest, but not externally citable, and I make no leaderboard claim. That's a deliberate boundary, not an oversight.

What I'd tell someone starting this

  • Build the measurement before the model. A decode bug once made every score read as zero; floor-adjusted evaluation against a known-good reference caught it.
  • Verify before you build. The distribution diagnosis and the MLX port were both confirmed cheaply before committing real compute.
  • Distrust single runs. Seed everything; report medians across seeds.
  • Write the negatives down. The failures, diagnosed, were more valuable than any single positive — they're what moved the project forward.
