# Observation: Adding MCQ Distill Data Regressed POPE Without Gaining MMBench (P2-B1)

**Date:** 2026-06-16
**Context:** P2-B1. The constructed student had real POPE grounding (balanced-acc 68.3) but was **at the MMBench majority floor** (0.44 ≈ 0.50) — no multiple-choice ability, because it never trained on the MCQ format. This run added teacher-generated MCQ items to the distill mix to try to lift MMBench off the floor.
**Verdict:** **Negative (task interference).** Mixing 491 MCQ items into the distill data **did not move MMBench off the floor** *and* **regressed POPE from real signal to below the chance floor.** All metrics here are floor-adjusted and internal-only.

---

## Setup

- Same recipe as the prior scale-up (align 3000 on `coco_caption_5k`, converged — align loss 1.81), but the distill set was the **combined `qa_balanced_mcq` cache**: 1386 balanced yes/no+open records **+ 491 teacher MCQ items** (4-option questions about COCO images, answer-letter format matching the eval). Distill 1500 steps (final loss 0.48), batch-1.
- MCQ data built with the new `services/distillation_pipeline.py --mode mcq`.
- Eval: floor-adjusted, same-path, n=100 (balanced accuracy for POPE; accuracy vs the majority floor for MCQ).

## Result (floor-adjusted)

| Benchmark | no-MCQ student | **+ MCQ data** | chance floor | verdict |
|---|---:|---:|---:|---|
| **POPE** (balanced-acc) | **68.3** ✓ above | **48.3** | 50 | **REGRESSED — now below floor** |
| **MMBench** (accuracy) | 0.44 (at floor) | **0.50** | 0.50 | still **at floor** (no gain) |
| RealWorldQA (accuracy) | 0.41 | 0.43 | 0.47 | uninformative at n=100 |

Two clear negatives: MMBench did **not** clear its floor, and POPE — the one axis with real signal — **fell to chance**.

## Why — task interference / forgetting (again)

This is the same failure family the project keeps hitting when distilling multiple skills into a tiny model with limited data/budget:
- **P2-D1:** caption distillation hurt MCQ grounding.
- **P2-D2:** presence-biased Q&A collapsed POPE to always-"Yes".
- **here:** MCQ data (a very different output format — emit a single letter A–D) pulled the model away from the yes/no grounding it had learned, **degrading POPE**, while **491 MCQ items in 1500 steps were too few to actually learn multiple-choice** (MMBench stayed at the floor).

Two compounding factors:
1. **Budget/forgetting.** Same 1500 distill steps now split across a new, format-divergent task → the model fits neither well and loses the old skill (no rehearsal protection at this budget).
2. **Distribution mismatch.** The MCQ items are about everyday COCO scenes; MMBench questions are largely knowledge/reasoning over diagrams and passages. COCO-image MCQs may not transfer to MMBench's distribution even if learned.

## What the system does next

P2-B1 stays **IN_PROGRESS**. The MCQ-data-as-mixed-distill lever (small pilot, same budget) is a **negative result** — don't repeat it as-is. Refined levers:
1. **Protect the existing skill** — rehearsal weighting / more distill steps so POPE is retained while MCQ is added (the P2-D2 rehearsal lesson, applied to format-mixing).
2. **Much more MCQ data** — 491 items is far too few to learn a new output format.
3. **Distribution-matched MCQ** — generate MCQs that resemble the target benchmarks (reasoning/knowledge), not just COCO object questions; or accept that COCO-trained students won't do MMBench and pick a benchmark the training distribution actually supports.
4. **Best student so far remains the no-MCQ scale-up** (`d3423bc0`): POPE balanced-acc 68.3 (real), MMBench at floor.

## Caveats

Benchmark numbers are **internal-only** (n=100 slice, non-official protocol, never reproduced a published number — see memory `benchmark-eval-internal-only`). The comparison here is **internally consistent** (same eval, floor-adjusted) and trustworthy for *this* A/B (MCQ-mix vs not), but not externally citable.
